# Focus Engine

一套帮人从"脑子里很多想法"收敛到"今天专注做这一件"的决策系统。

## 核心信念（为什么这样设计）

这个 skill 的价值不在"排序"，而在"砍"和"收敛"。大部分人的问题不是不知道怎么排列十件事，而是十件事里大部分现在根本不该做，却都在抢注意力。所以系统的首要功能是把它们降到一个 Priority One，并给被砍掉的想法一个明确的"暂时不做"的理由，让它别再反复回来骚扰用户。

两条铁律，违背就会退化成"剧场化"（看起来在判断、其实在拍脑袋）：

1. **没有事实，不做判断。** 任何排序之前，必须先采集两层事实：每个想法的真实现状（做到哪一步了、有没有人付钱、数据如何），以及外部市场事实（热点在哪、竞争多激烈、缝在哪）。跳过这步直接套框架，是这个系统最大的失败模式。
2. **判断逻辑写死在框架里，不靠临场发挥。** 下面的三个维度、过滤顺序是固定的。Claude 的工作是把用户的具体想法套进框架回答，不是每次重新发明判断标准。

## 工作流：OODA 四步

每天早上跑一次。骨架是 OODA（Observe → Orient → Decide → Act），对应四个动作。不是每次都从头跑——见下方"每日 vs 每周"。

### 1. OBSERVE — 采集事实（最重要、最容易被跳过的一步）

分两半：

**内部现状 intake。** 对每个想法/项目，确认以下字段。缺失的就直接问用户，不要自己脑补填上去——脑补现状是头号失败模式。

- 它是什么？目标是什么（赚钱？涨粉？验证？）
- 现在做到哪一步了？（纯想法 / 有雏形 / 已上线 / 已有用户）
- 离"有人付钱"还差几步？
- 有任何真实数据吗？（访客、注册、付费、粉丝、互动）
- 当前最卡的地方是什么？

把结果填进 `references/intake-template.md` 的表格形式呈现给用户，标出缺口。**只要还有影响判断的关键缺口没填，就停下来问，不要往下走。**

**外部市场事实 market-scan。** 当判断涉及"这个方向值不值得做""热点在哪""竞争多激烈""哪里是 less competitive market"时，用 web_search 实际去查，不要凭记忆答。要查的四件事见 `references/market-scan.md`。市调结果常常会直接改变对某个赌注的判断（例如发现某赛道已红海，但某个细分缝无人做）。

### 2. ORIENT — 套框架定向

把采集到的事实，过三道**独立**的问。不要合成一个加权总分——加权总分会系统性地砍错"暂时不赚钱但反哺全局"的项目（这正是 RICE/ICE 的缺陷）。三道问分别答：

1. **离钱距离** — 这步推进后，离"有人付钱"近了几格？
2. **飞轮杠杆** — 它解锁/反哺了几个其他项目？是不是当前的瓶颈节点？（瓶颈理论：优化非瓶颈等于零）
3. **资产复利** — 做完它，沉淀下可复用的技能/工具/内容资产吗？

何时调用更重的模型，见 `references/decision-models.md`（杠铃策略、瓶颈理论、Cynefin、Pretotyping、OODA 等）。不是每天全用，按需取。

定向的关键产物是判断**当前瓶颈在哪**——整个飞轮里最卡的那一个节点。Priority One 应该指向瓶颈，而不是指向最舒服或最赚钱的节点。

### 3. DECIDE — 收敛出 Priority One

强制收敛到**一个**今日主攻 + 最多两个顺手小动作。输出用 `assets/daily-output-template.md` 的固定结构，禁止偏离：

```
## 今日主攻 (Priority One)
- 具体动作：[具体到能立刻动手，不是"做CJC"而是"改CJC hero区文案，突出交易所insider背景"]
- 为什么是它：[命中哪个维度 / 指向哪个瓶颈]
- 完成判定：[一个明确的终点，避免没有终点的伪任务]

## 顺势推飞轮（≤2个，可选）
- [只放能利用碎片时间、且反哺主攻的小动作]

## 今日不做 / 推迟（附理由）
- [项目]：[为什么今天不做，推迟到什么条件满足]
```

收敛规则见 `references/decision-models.md` 的"去繁就简"段。核心：当两个候选难分时，选离瓶颈更近的；瓶颈相同则选离钱更近的。

### 4. ACT — 交付

把上面的输出给用户。不要附加冗长解释。如果用户当天回顾（晚上），用 `review` 逻辑更新现状、记录哪些被砍的理由是否成立，为明天的 intake 做准备。

## 每日 vs 每周

- **每周一次**（建议周一）：重新跑完整 OBSERVE（含 market-scan）+ ORIENT，重新判断瓶颈在哪、主赌注是否要换。瓶颈会移动，但不该每天换。
- **工作日每天早上**：在既定主赌注框架下，只跑轻量 intake（昨天有什么变化）+ DECIDE，产出当天 Priority One。不重排全局——频繁重排本身是一种逃避执行。

## 多用户适配

判断框架（三维度、OODA 骨架）是通用的，跨用户不变。变的是用户的具体项目清单和约束——这些放在 `references/intake-template.md`，用户首次使用时填一份自己的 `bets.md`（当前所有项目+状态）和 `constraints.md`（runway、目标、每周可投入时间）。
