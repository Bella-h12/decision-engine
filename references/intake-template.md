# 现状采集模板

## 单个项目现状表（每个想法都填一行）

| 字段 | 说明 | 缺失时怎么办 |
|---|---|---|
| 名称 | 项目/想法叫什么 | — |
| 目标 | 赚钱 / 涨粉 / 验证假设 / 积累资产 | 问用户，目标不清就无法判断离钱距离 |
| 当前阶段 | 纯想法 / 有雏形 / 已上线 / 已有用户 / 已有营收 | 问用户，禁止脑补 |
| 离钱距离 | 离"有人付钱"还差几步，列出来 | 由阶段推导，但要跟用户确认 |
| 真实数据 | 访客/注册/付费/粉丝/互动的具体数字 | **必问**，没有数字的判断都是猜 |
| 当前瓶颈 | 这个项目现在最卡在哪 | 问用户 |
| 依赖/反哺 | 它依赖哪些项目？又反哺哪些项目？ | 用来画飞轮图 |

## 采集纪律

1. **缺口优先于推进。** 只要还有影响判断的关键字段为空（尤其"真实数据"和"当前阶段"），就停下来问，不要带着脑补的数据往下排序。
2. **区分"想做"和"在做"。** 用户嘴上说的优先级，和项目真实进度常常不符。以现状为准。
3. **把所有项目画成一张依赖图**，而不是一个列表。项目之间往往互相反哺（做内容→涨粉→给产品导流），列表会丢掉这层结构，导致砍错。

## 用户档案模板（首次使用时各填一份）

### bets.md（当前所有项目 + 状态）

```
# 我的项目清单

## 项目1：[名称]
- 目标：
- 当前阶段：
- 真实数据：
- 瓶颈：
- 反哺谁 / 被谁反哺：

## 项目2：...
```

### constraints.md（硬约束）

```
# 我的约束

- Runway（还能撑多久）：
- 首要目标：
- 每周可投入时间：
- 不可逾越的红线（比如某些事绝不做）：
```

这两份文件是"多用户适配"的落点——框架不变，档案因人而异。每周重评时更新一次。
